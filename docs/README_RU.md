# TCP KCC v1.0 (Управление перегрузкой Калмана)

Модуль управления перегрузкой TCP для VPS-сред с общей полосой пропускания, объединяющий конечный автомат BBRv1 с фильтром Калмана для оценки задержки распространения.

## Принципы проектирования

Алгоритмы управления перегрузкой должны балансировать между пропускной способностью, задержкой, справедливостью и устойчивостью к потерям. KCC использует прагматичный подход:

1. BBRv1 предоставляет проверенную основу. Конечный автомат, пакетирование, циклические усиления, STARTUP/DRAIN/PROBE_BW/PROBE_RTT — KCC заимствует эти механизмы без изменений.

2. Фильтр Калмана повышает точность оценки. Разделение истинной задержки распространения от задержки в очереди и джиттера позволяет получить более точную оценку min_rtt, что ведёт к более точному расчёту BDP, лучшей калибровке CWND и более стабильному пакетированию.

3. Межалгоритмическая динамика следует стандартному конкурентному равновесию TCP. KCC не ограничивает искусственно свою скорость отправки в ответ на очередь, обнаруженную от внешних потоков. Затухание усиления (уменьшение зондирования на основе очереди) доступно как опция через kcc_cycle_decay_mask, но отключено по умолчанию для сохранения полной интенсивности зондирования.

4. Внутри-KCC справедливость активно поддерживается. Сходимость Калмана гарантирует, что потоки KCC на одном хосте используют согласованную оценку min_rtt, устраняя петлю обратной связи «победитель получает всё», которая вызывает серьёзную несправедливость в чистых многопоточных развёртываниях BBR.

## Обзор алгоритма

TCP KCC реализует модуль управления перегрузкой на стороне отправителя для ядра Linux в виде загружаемого `tcp_kcc.ko`. Функция управления перегрузкой `kcc_main()` вызывается при каждом ACK из `tcp_ack()`, получая структуру `rate_sample`, содержащую оценки пропускной способности и RTT на уровне ядра, а также счетчики доставки и потерь. Алгоритм работает в двух временных режимах: **быстрый путь на каждый ACK**, который обновляет состояние измерений и вычисляет мгновенные цели pacing и окна, и **медленный путь на каждый раунд**, который оценивает условия перехода состояний и пересчитывает коэффициенты усиления.

Основной конвейер измерений состоит из двух компонентов:

1. **Фильтр максимальной пропускной способности со скользящим окном** (`minmax_running_max` из `linux/win_minmax.h`): окно охватывает последние `kcc_bw_rt_cycle_len` (по умолчанию 10) кругов. Предоставляет оценку `max_bw`, совместимую с BBR.

2. **Оценщик задержки распространения на фильтре Калмана**: заменяет скользящее окно минимального RTT из BBRv1, и является источником по умолчанию для оценки RTT в BDP (см. [Стратегия RTT Модели](#стратегия-rtt-модели)). Одномерный фильтр Калмана (Kalman 1960), работающий в единицах с фиксированной запятой `kcc_kalman_scale` × мкс, моделирующий истинную задержку распространения как случайное блуждание:
   - Состояние: `x[k] = x[k−1] + w[k]`, `w ~ N(0, Q)`
   - Наблюдение: `z[k] = x[k] + v[k]`, `v ~ N(0, R)`

Соглашения фиксированной запятой: `BW_UNIT = 1 << 24` для пропускной способности (сегменты * 2^24 / мкс), `BBR_UNIT = 1 << 8 = 256` как безразмерная единица усиления.

## Стратегия RTT Модели

KCC предоставляет настраиваемую стратегию выбора оценки RTT для расчёта BDP (Bandwidth-Delay Product), управляемую через `kcc_rtt_mode`:

| Режим | Значение | Поведение | Применение |
|-------|----------|-----------|------------|
| FILTER | 1 (по умолчанию) | Прямое использование `x_est_us` — сырой оценки фильтра Калмана/скользящего окна | Продакшн WAN/VPS: устойчивость к смене маршрутов, отсутствие провалов пропускной способности |
| MIN | 0 | `min(x_est_us, min_rtt_us)` — ограничение оценки Калмана оконным минимумом | Проверка стабильности модуля ядра; каналы с фиксированным RTT |

**Почему FILTER используется по умолчанию:**

- **Устойчивость к смене маршрутов**: При изменении BGP-маршрута, увеличивающем физический RTT (например, 50 мс → 100 мс), коэффициент усиления Калмана K_k реагирует в течение нескольких RTT и подтягивает оценку к новой задержке. Режим MIN блокируется на старом `min_rtt_us`, уменьшая BDP вдвое.

- **Встроенная защита**: Гейтинг выбросов отбрасывает выборки с пиками очереди до их попадания в фильтр. Адаптивная оценка шума Q/R снижает усиление Калмана при высокой зашумлённости сети, поэтому фильтр естественным образом не доверяет кратковременному разбуханию очереди и удерживает оценку вблизи истинной задержки распространения.

- **Отвязка PROBE_RTT**: Режим FILTER включает функцию `kcc_probe_rtt_decouple` — фильтр Калмана отслеживает минимальный RTT без необходимости периодического сброса до 4 пакетов.

Переключение во время выполнения: `echo 0 > /proc/sys/net/kcc/kcc_rtt_mode` для возврата в режим MIN.

## Конечный автомат

```
    ┌───> STARTUP ────┐
    │       │         │
    │       ▼         │
    │     DRAIN  ─────┤
    │       │         │
    │       ▼         │
    └─── PROBE_BW ────┘
    │      ^    │
    │      │    │
    │      └────┘
    │
    └─── PROBE_RTT <──┘
```

Четыре режима, закодированные в 2-битном поле `mode` в `struct KCC`:

- **STARTUP (0)**: Начальное состояние. pacing_gain ≈ 2.885x (`kcc_high_gain_val`), cwnd_gain также 2.885x. Экспоненциальный поиск пропускной способности.
- **DRAIN (1)**: Вход после выхода из STARTUP. pacing_gain ≈ 0.347x (`kcc_drain_gain_val`), cwnd_gain остается 2.885x. Опустошение очереди, накопленной во время STARTUP.
- **PROBE_BW (2)**: Установившееся состояние. Циклически проходит таблицу усилений из 256 слотов (повторяющийся шаблон 8 фаз: 1.25x/0.75x/8×1.0x).
- **PROBE_RTT (3)**: Периодически уменьшает количество пакетов в полете до `kcc_cwnd_min_target` (по умолчанию 4 сегмента) для получения свежего отсчета RTT.

### STARTUP → DRAIN

Срабатывает, когда установлен `full_bw_reached` — после `kcc_full_bw_cnt` (по умолчанию 3) последовательных раундов, в которых `max_bw` не увеличивается как минимум на `kcc_full_bw_thresh_val` (по умолчанию 1.25x) по сравнению с ранее наблюдавшимся пиком. BDP с коэффициентом 1.0x записывается в `snd_ssthresh`. `qdelay_avg` сбрасывается в ноль, чтобы предотвратить влияние накопленной очереди STARTUP на PROBE_BW.

### DRAIN → PROBE_BW

Срабатывает, когда расчетный объем в полете при EDT ≤ целевой объем в полете при усилении BDP 1.0x. **Оптимизация пропуска DRAIN**: когда фильтр Калмана сошелся И `qdelay_avg` ниже `kcc_drain_skip_qdelay_us` (по умолчанию 1000 мкс), фаза DRAIN пропускается — происходит ранний переход в PROBE_BW.

При входе в PROBE_BW индекс фазы цикла рандомизируется: `cycle_idx = len − 1 − rand(kcc_probe_bw_cycle_rand)` (по умолчанию `len − 1 − rand(8)`), что декоррелирует параллельные потоки, разделяющие узкое место.

### PROBE_BW → PROBE_RTT

Срабатывает, когда истекает интервал фильтра PROBE_RTT — отметка времени `min_rtt_stamp` не обновлялась в течение вычисленного интервала. cwnd сохраняется в `prior_cwnd`, pacing устанавливается на опорожнение.

### PROBE_RTT → PROBE_BW

После того как объем в полете падает до `kcc_cwnd_min_target` или наблюдается граница раунда, состояние сохраняется как минимум `kcc_probe_rtt_mode_ms_val` (по умолчанию 200 мс) и после завершения как минимум одного полного раунда, затем происходит выход. cwnd восстанавливается до `prior_cwnd`, затем входит в стандартный цикл усиления PROBE_BW.

### Восстановление и потери

- При TCP_CA_Loss: `full_bw` и `full_bw_cnt` сбрасываются, `round_start` устанавливается в 1, `packet_conservation` очищается до 0.
- Вход в восстановление (TCP_CA_Recovery): `packet_conservation` включается, cwnd = inflight + acked.
- Выход из восстановления: восстанавливается до `prior_cwnd`, `packet_conservation` очищается.
- `kcc_undo_cwnd()`: сбрасывает `full_bw` и `full_bw_cnt` (сохраняя `full_bw_reached`), очищает состояние LT BW.

### Обнаружение раундов (BBR)

`next_rtt_delivered` инициализируется в `0` (совместимость со стандартным BBR; Cardwell et al. 2016), поэтому первый ACK немедленно начинает обнаружение раунда 1 без искусственного смещения. Граница раунда обнаруживается, когда `prior_delivered >= next_rtt_delivered`, с защитой `interval_us <= 0` (соответствует BBR `delivered < 0 || interval_us <= 0`) — перехватывает нулевые и отрицательные интервалы, предотвращая их попадание в измерительный конвейер.

- `next_rtt_delivered` инициализируется в `0` (совместимость с BBR): первый ACK начинает раунд 1.
- `interval_us <= 0` проверка: полное соответствие BBR, отклоняет отрицательные интервалы.
- `round_start` сбрасывается в `0` в начале `kcc_update_bw()`, до проверки валидности — соответствует позиции `bbr->round_start = 0` в BBR.

## Основные измерения

### Оценка пропускной способности

Фильтр максимальной пропускной способности со скользящим окном (`minmax_running_max` из `linux/win_minmax.h`) за `kcc_bw_rt_cycle_len` (по умолчанию 10) раундов. Мгновенная bw = `delivered × BW_UNIT / interval_us` вычисляется на каждый ACK. Подается в скользящее окно только когда не ограничено приложением или когда bw ≥ текущего максимума (правило BBR).

Когда активен `lt_use_bw`, оценка активной пропускной способности переключается на `lt_bw` (долгосрочная оценка пропускной способности).

### Фильтр Калмана

Одномерная скалярная рекурсия Калмана (сложность O(1)):

```
Предсказание:
  x_pred = x_est          (тождественный переход состояния)
  p_pred = p_est + Q      (предсказание ковариации)

Обновление:
  innov   = z − x_pred    (инновация)
  K       = p_pred / (p_pred + R)   (коэффициент Калмана [0,1])
  x_est   = x_pred + K × innov      (обновление состояния)
  p_est   = (1 − K) × p_pred        (апостериорная ковариация)
```

**Адаптивный шум процесса Q**:
```
Q_base   = kcc_kalman_q (по умолчанию 100)
q_factor = max(kcc_kalman_q_min_factor_val, min_rtt_us / kcc_kalman_q_rtt_div)
Q        = min(Q_base × q_factor, Q_base × kcc_kalman_q_scale_cap)
Q        = min(Q, kcc_kalman_q_max)
```

**Адаптивный шум измерения R**:
```
R = R_base + max(0, jitter_ewma − kcc_jitter_r_thresh_us) × R_base / kcc_jitter_r_scale
R = min(R, R_base × kcc_kalman_r_max_boost)
```

**Q-Boost обнаружение смены пути (с порогом уверенности + перезарядка)**: когда `|innovation| > kcc_kalman_q_boost_thresh_val` (по умолчанию ≈ 4 мс сдвиг RTT) И фильтр сошелся (`p_est ≤ kcc_kalman_converged_p_est_val`, по умолчанию 500), `p_est` сбрасывается на `kcc_kalman_p_est_init_val`, увеличивая коэффициент Калмана к 1.0 для быстрой сходимости. Период перезарядки `kcc_kalman_qboost_cdwn` (по умолчанию 15) выборок между последовательными событиями qboost предотвращает неконтролируемое срабатывание на каналах с потерями и высоким джиттером RTT.

**Шлюз выбросов**: динамический порог `dyn_thresh = max(outlier_ms × 1000 × scale, jitter_ewma × outlier_jitter_mult × scale)`. Применяется только когда `p_pred ≤ kcc_kalman_converged_p_est_val`. После `kcc_kalman_max_consec_reject` (по умолчанию 25) последовательных отклонений следующий образец принудительно принимается, чтобы предотвратить самоблокировку.

**Оценка шума, согласованная с ковариацией (BBR-S)**: `q_est = (1−α) × q_est + α × (K × innov)²`, `r_est = (1−β) × r_est + β × max(0, innov² − p_pred)`. Режим комбинации: режим 0 = только эвристический, режим 1 = максимум (по умолчанию), режим 2 = взвешенная смесь.

**Переход Калмана**: когда `x_est > 0` и `sample_cnt ≥ kcc_kalman_min_samples` (по умолчанию 5), `min_rtt_us` заменяется на `x_est / kcc_kalman_scale`. `min_rtt_stamp` не обновляется — триггер интервала PROBE_RTT остается независимым.

**Стратегия RTT модели**: Оценка RTT для расчёта BDP контролируется `kcc_rtt_mode`. В режиме FILTER (по умолчанию) `model_rtt = x_est_us` напрямую — оценка Калмана/скользящего окна используется без ограничений. В режиме MIN `model_rtt = min(x_est_us, min_rtt_us)` — оценка Калмана ограничивается оконным минимумом, гарантируя, что BDP никогда не завышается. Режим FILTER по умолчанию рекомендуется для продакшн WAN/VPS, где задержка канала может изменяться внезапно (смена BGP-маршрутов, хэндоверы LEO, переключение сотовых вышек). См. [Стратегия RTT Модели](#стратегия-rtt-модели).

## Улучшения BBR

### Затухание усиления

Включено 256-битной битовой маской `kcc_cycle_decay_mask[]` для определенных фаз PROBE_BW. Формула затухания (при принятом образце Калмана):

```
max_red       = probe_gain − BBR_UNIT
conf_scale    = обратное масштабирование p_est (BBR_UNIT при полной уверенности)
qdelay_decay  = min(max(0, qdelay_avg − qthresh) × BBR_UNIT / qscale, max_red)
                    × conf_scale / BBR_UNIT
jitter_decay  = min(max(0, jitter_ewma − jthresh) × BBR_UNIT / jscale, remaining)
                    × conf_scale / BBR_UNIT
effective     = max(probe_gain − qdelay_decay − jitter_decay, BBR_UNIT)
```

Масштабирование уверенности Калмана: когда `p_est > kcc_kalman_converged_p_est`, затухание пропорционально уменьшается, избегая излишнего снижения, когда фильтр не уверен.

### ECN Backoff

Условия активации (все должны выполняться):
1. `kcc_ecn_enable_val != 0`
2. Фильтр Калмана сошелся (`p_est < converged`, `sample_cnt >= min_samples`)
3. `ecn_ewma > 0` (наблюдались CE-метки)
4. `qdelay_avg > kcc_ecn_qdelay_thresh_us_val` (по умолчанию 2000 мкс)
5. Режим НЕ PROBE_BW (cwnd_gain фиксирован на 2x в PROBE_BW)

Во время фаз зондирования (`pacing_gain > BBR_UNIT`), ECN backoff градуируется по `BBR_UNIT² / pacing_gain` — ~80% снижения при зондировании 1.25x, ~65% при усилении STARTUP 2.89x.

EWMA коэффициента ECN-меток: обновляется на границах раундов по `kcc_ecn_ewma_retained / kcc_ecn_ewma_total` (по умолчанию 3/4), с плавным затуханием на каждый ACK по `kcc_ecn_idle_decay_num / kcc_ecn_idle_decay_den` (по умолчанию 31/32) на каждый ACK без новых CE-меток.

### Обнаружение Одиночного Потока

Когда KCC обнаруживает, что поток, вероятно, один на узком месте (низкая задержка очереди, низкий джиттер, отсутствие меток ECN, отсутствие агрегации ACK), он автоматически переходит в чистый режим BBR:

- `kcc_get_model_rtt()` возвращает `min_rtt_us` напрямую (в обход сглаженной оценки Калмана, которая имеет небольшое положительное смещение из-за одностороннего шума измерений).
- `kcc_ecn_backoff()` настраивается через `kcc_alone_bypass_ecn` (по умолчанию 1) — на пути с одним потоком метки ECN являются ложными срабатываниями AQM, поскольку нет конкурирующего отправителя. Пропуск соответствует нулевому поведению ECN в BBR. Установите 0, чтобы сохранить откат ECN даже в одиночном режиме (консервативный).

Это устраняет разрыв в пропускной способности одиночного потока между KCC и BBR, сохраняя при этом полный защитный контур KCC (Калман, откат ECN, затухание усиления) для сценариев с несколькими потоками.

**Гистерезис**: для входа требуется `kcc_alone_confirm_rounds` (по умолчанию 3) последовательных подходящих раундов——предотвращает колебания во время коротких тихих периодов при конкуренции нескольких потоков ("консервативное ускорение"). Выход: во время оценки в фазе cruise любой одиночный провал квалификации сбрасывает флаг ("агрессивное торможение").

**Компромисс проектирования**: Потеря пакетов НЕ используется в качестве дисквалификатора одиночного потока — некоторые каналы (мелкие буферы, беспроводные сети, burst drop при виртуализации) имеют врожденные потери, не связанные с конкуренцией. Приравнивание потерь к многопоточной конкуренции вызывает колебания на путях с одним потоком. Сигнал LT BW (обнаружение полисера в BBR по потерям) не участвует в оценке одиночного потока.

**Гейтирование усиления**: оценка одиночного потока выполняется только во время фазы cruise (`pacing_gain == BBR_UNIT`). Probe-up (1.25x) намеренно давит на узкое место — его давление на очередь является самоиндуцированным, а не сигналом конкуренции. Drain (0.75x) искусственно подавляет очередь. Ограничивая оценку cruise (равновесным стационарным состоянием), самоиндуцированное давление probe-up больше не вызывает ложных выходов из одиночного режима.

Условия квалификации (все пять должны выполняться на границе раунда):
0. Калман сошёлся (`sample_cnt >= kcc_kalman_min_samples`) — доверять qdelay/jitter как сигналам очереди
1. `qdelay_avg < kcc_alone_qdelay_thresh_us` (по умолчанию 1000 us) — очередь почти пуста
2. `jitter_ewma < kcc_alone_jitter_thresh_us` (по умолчанию 2000 us) — только микроджиттер тактов ACK
3. `ecn_ewma == 0` — отсутствие меток перегрузки от AQM
4. `agg_state <= max` согласно `kcc_alone_agg_state_level` (по умолчанию 1) — три уровня строгости агрегации ACK: 0 = только IDLE (самый строгий, нулевая агрегация), 1 = ≤ SUSPECTED (по умолчанию, допускает временную агрегацию), 2 = ≤ CONFIRMED (самый разрешительный, блокирует только постоянную агрегацию)

### Динамический интервал PROBE_RTT

Отображает `p_est` Калмана в интервал PROBE_RTT для каждого соединения:

```
p_est ≤ converged:              interval = dyn_max (по умолчанию 30 с)
p_est ≥ high (= mult × conv):   interval = base (по умолчанию 10 с)
converged < p_est < high:       линейная интерполяция
```

Уменьшает частоту PROBE_RTT при высокой уверенности (низкий `p_est`), снижая джиттер пропускной способности на стабильных путях. Возвращается к классическому 10-секундному интервалу при низкой уверенности.

**Джиттер входа для каждого потока**: Чтобы предотвратить одновременный вход всех сосуществующих потоков в PROBE_RTT (сброс до 4 пакетов ~1.8 Mbps, затем заполнение с коэффициентом 2.89×), каждый поток добавляет хэш-производный джиттер (разброс 0–845 мс) к своему интервалу PROBE_RTT. В любой момент времени в PROBE_RTT находится максимум ~1 поток, что устраняет одновременный коллапс сброса/заполнения, вызывающий RTO.

### Отвязка PROBE_RTT и Умная Рекалибровка

Механизм PROBE_RTT в BBRv1 каждые ~10 секунд опустошает канал до 4 пакетов для измерения `min_rtt_us`. Это необходимо для оконного оценщика минимального RTT — окно не может отличить задержку распространения от задержки очереди, пока канал не пуст. Цена — периодический провал пропускной способности («пила» BBR).

В режиме FILTER фильтр Калмана полностью заменяет окно. Он может отделить шум очереди от истинной задержки распространения через гейтинг выбросов и адаптивную оценку шума — опустошение канала не требуется. Параметр `kcc_probe_rtt_decouple` (по умолчанию 1) управляет этим:

| Режим | Значение | Поведение |
|-------|----------|-----------|
| Отвязан | 1 (по умолчанию) | **Калман здоров** (p_est ≤ `kcc_recal_p_est_thresh`): полностью подавить PROBE_RTT → нулевые провалы пропускной способности, нулевые синхронные коллапсы. **Калман разошёлся** (p_est > порог): автоматически запустить традиционный PROBE_RTT как страховочную сеть → восстановить базовый уровень фильтра, отвязка возобновляется автоматически. |
| Традиционный | 0 | Слепой периодический PROBE_RTT каждые ~10 с (совместимость с BBR). |

**Эвристика умной рекалибровки** (`kcc_kalman_needs_recalibration()`): В установившемся режиме на стабильном канале ковариация ошибки Калмана p_est сходится к p_est_floor (~4–10), что на порядки ниже порога `kcc_recal_p_est_thresh` (250 000 = 25% от p_est_max). Рост p_est сигнализирует, что внутренняя модель фильтра больше не объясняет наблюдения — обычно потому, что канал существенно изменился. Когда p_est превышает порог, один традиционный сброс PROBE_RTT восстанавливает базовый уровень фильтра; Калман заново сходится, и отвязка возобновляется автоматически.

Это превращает PROBE_RTT из **слепого периодического самоповреждения** в **интеллектуальную рекалибровку на основе доверия** — протокол опустошает канал только тогда, когда имеет эмпирические доказательства потери фильтром уверенности.

Требуется `kcc_rtt_mode == 1`. Не действует в режиме MIN (режим MIN зависит от PROBE_RTT для обновления `min_rtt_us`).

| Параметр | По умолчанию | Диапазон | Описание |
|----------|--------------|----------|----------|
| `kcc_probe_rtt_decouple` | 1 | 0–1 | Включить отвязку PROBE_RTT (только режим FILTER) |
| `kcc_recal_p_est_thresh` | 250 000 | 1–100 000 000 | Порог p_est для запуска страховочной рекалибровки |

### Долгосрочная оценка пропускной способности (LT Bandwidth)

Оценщик нижней границы, запускаемый потерями. Интервал выборки охватывает [4, 16] RTT. Действителен, когда коэффициент потерь ≥ 5.9% (`kcc_lt_loss_thresh` по умолчанию 15/256). Пропускная способность `bw = delivered × BW_UNIT / interval_us`.

В отличие от простого среднего BBR (`(bw + lt_bw) >> 1`), KCC использует настраиваемую EMA (`kcc_lt_bw_ema_num / kcc_lt_bw_ema_den`, по умолчанию 1/2 = 0.5):

```
lt_bw = (bw_new × en + lt_bw × (ed − en)) / ed
```

Активация отличается от BBR: KCC сохраняет `lt_bw` при первом действительном интервале, но НЕ устанавливает `lt_use_bw`; требуется согласованность с предыдущим интервалом — уменьшает ложную активацию из-за шума измерений.

**Двухпороговый шлюз перегрузки**: Перед установкой `lt_use_bw = 1` оцениваются как постоянная проверка очереди EWMA (`qdelay_avg > kcc_ecn_qdelay_thresh_us_val`), ТАК и мгновенная проверка очереди на основе SRTT (`srtt_us − min_rtt_us > kcc_lt_bw_inst_qdelay_thresh_us`, по умолчанию 5000 мкс). При обнаружении перегрузки выборка LT BW прерывается. Проверка SRTT работает без выделения `ext`, обеспечивая защиту от сбоев выделения.



### Компенсация агрегации ACK на основе уверенности (вдохновлено BBRplus)

Добавляет второй уровень с порогом уверенности поверх традиционного двухслотового оценщика extra-acked.

**Четыре ортогональных фактора** (каждый вносит `kcc_agg_factor_weight` очков, по умолчанию 256):
1. Фильтр Калмана сошелся (`p_est < converged` + `sample_cnt >= min_samples`)
2. Не в состоянии восстановления потерь (`icsk_ca_state < TCP_CA_Recovery`)
3. RTT в пределах `min_rtt_us + kcc_agg_factor3_qdelay_us` (по умолчанию 2 мс) от истинной задержки распространения
4. `extra_acked` в пределах `kcc_agg_factor4_ratio_num/den` (по умолчанию 1.5x) от максимума окна

**Четыре состояния**: IDLE (< `kcc_agg_thresh_suspected`=256), SUSPECTED (≥256), CONFIRMED (≥512), TRUSTED (≥768).

**Сигнальный уровень** (всегда активен): уверенность линейно интерполирует коэффициент масштабирования R `[r_min, r_max]`. R возрастает мгновенно (быстрый отклик), затухает со скоростью `kcc_agg_r_hysteresis`% (по умолчанию сохраняется 75%, ~4 RTT до базового уровня) за RTT.

**Уровень управления** (`agg_state ≥ CONFIRMED`): пятиуровневая компенсация cwnd с защитой:
1. Блокируется, если задержка в очереди > `kcc_agg_safety_qdelay_us` (по умолчанию 4 мс)
2. Блокируется во время восстановления потерь
3. Блокируется, если cwnd > `BDP × kcc_agg_safety_bdp_mult` (по умолчанию 3x)
4. Блокируется, если объем в полете > безопасный cwnd + цель TSO сегментов
5. Сторожевой таймер: понижает CONFIRMED→SUSPECTED после `kcc_agg_max_comp_duration` (по умолчанию 8) последовательных RTT

### Сброс qdelay_avg при DRAIN

При переходе в DRAIN, `qdelay_avg` сбрасывается в ноль, предотвращая сохранение оценки очереди из STARTUP в PROBE_BW.

### Адаптация делителя TSO

`kcc_min_tso_segs()` корректирует пороговый делитель скорости на основе состояния Калмана:
- Фильтр Калмана сошелся + `jitter_ewma < 1000 мкс`: делитель уменьшается вдвое (8→4), более крупные пакеты TSO
- `jitter_ewma > 4000 мкс`: делитель удваивается (8→16), более мелкие пакеты TSO для подавления джиттера

## Скорость пакетирования и Cwnd

### Скорость пакетирования

```
rate = bw × mss × pacing_gain >> BBR_SCALE      // регулировка усиления
rate = rate × USEC_PER_SEC >> BW_SCALE            // преобразование в байты/с
rate = rate × margin_div / 100                    // запас пакетирования (по умолчанию 1%, matching BBR)
```

Изменения скорости применяются немедленно (без сглаживания), как в BBR (Cardwell et al. 2016). После `full_bw_reached`: все изменения скорости записываются немедленно. В STARTUP/DRAIN: применяются только увеличения (`rate > sk_pacing_rate`).

### Cwnd

```
target = BDP(bw, gain, ext)                       // базовый BDP
target = quantization_budget(target)              // запас TSO + выравнивание + бонус фазы 0
target += ack_agg_bonus + agg_compensation        // компенсация агрегации ACK

// прогрессия cwnd
if full_bw_reached:
    cwnd = min(cwnd + acked, target)              // сходимость к цели
else (STARTUP):
    cwnd = cwnd + acked                          // экспоненциальный рост

cwnd = max(cwnd, cwnd_min_target)                 // абсолютный минимум 4
Режим PROBE_RTT: cwnd = min(cwnd, cwnd_min_target) // минимальный объем в полете
```

## Внутренний поток фильтра Калмана

```
Отсчет RTT (rtt_us)
    │
    ├── Недействителен (≥0 и < dynamic_max)? Нет → отбросить
    │
    ├── Холодный старт (sample_cnt==0)? Да → инициализация: x_est=z, p_est=max(p_init, rtt_us/div)
    │                                           (обходит шлюз максимума RTT)
    │
    ├── Адаптивный Q: Q_base × max(q_min_factor, min_rtt_us / q_rtt_div)
    │   Адаптивный R: R_base + max(0, jitter − jr_thresh) × R_base / jr_scale
    │
    ├── Инновация: innov = z − x_est
    │
    ├── Q-Boost: |innov| > boost_thresh && p_est ≤ converged && время охлаждения истекло?
    │   ├── Yes: p_est = p_est_init, cooldown = 15, mark qboost_fired
    │   └── No:  cooldown-- if active
    │
    ├── Предсказание: p_pred = p_est + Q
    │
    ├── Шлюз выбросов: |innov| > dyn_thresh && p_pred ≤ converged?
    │   ├── Да и reject_cnt < max → отклонить, ++consec_reject_cnt, возврат
    │   └── Да и reject_cnt ≥ max → принудительно принять (антиблокировка)
    │
    └── Обновление Калмана:
         ├── K = p_pred / (p_pred + R)
         ├── x_est += K × innov (ограничено неотрицательным)
         ├── p_est = max(p_floor, (1 − K) × p_pred)
         ├── Обновление EWMA джиттера
         ├── Обновление EWMA qdelay
         ├── Оценка шума, согласованная с ковариацией BBR-S
         └── sample_cnt++
```

## Модель параллелизма и безопасности

KCC намеренно не использует READ_ONCE/WRITE_ONCE или RCU для своих собственных структур данных. Этот подход соответствует всем внутриядерным модулям CC, таким как BBR и CUBIC.

`kcc_init()` выполняется в контексте процесса (во время создания сокета), до того как сокет будет открыт для любого softirq. `kcc_release()` выполняется после того, как ядро гарантирует, что ни один softirq больше не обрабатывает ACK этого сокета. Временное устаревшее значение глобального параметра модуля влияет максимум на один ACK и исправляется при следующем ACK.

Единственное исключение: `sk->sk_pacing_rate` / `sk->sk_pacing_shift` — это поля уровня сокета, которые пользовательское пространство может одновременно изменять через `setsockopt`, поэтому сохраняется соглашение BBR о WRITE_ONCE/READ_ONCE.

## Global Kalman BDP — Межсоединительная Инжекция Пропускной Способности

KCC v1.0 включает опциональный межсоединительный Глобальный Фильтр Калмана, который оценивает установившуюся пропускную способность узкого места сервера. Эта оценка используется для запуска новых соединений на консервативно низкой "десертной скорости" — достаточно быстрой, чтобы пропустить холодный старт, и достаточно медленной, чтобы избежать перерегулирования.

### Принцип Проектирования

Фильтр получает выборки пропускной способности из **крейсерской фазы** PROBE\_BW (усиление = 1.0×) всех соединений KCC. Выборки крейсерской фазы являются самым чистым сигналом истинной доступной пропускной способности — без перерегулирования зондирования 1.25×, без недооценки дренажа 0.75×. Одномерный фильтр Калмана со случайным блужданием (Kalman 1960) отслеживает глобальное установившееся состояние.

При установлении нового соединения оценка фильтра используется для инициализации:

| Вводимое значение | Назначение |
|----------------|---------|
| `minmax` (max\_bw tracker) | Инициализация истории пропускной способности скользящего окна, чтобы первые грязные выборки ACK не утянули её в ноль |
| `sk_pacing_rate` | Начальная скорость пакетирования при нейтральном усилении (BBR\_UNIT); усиление STARTUP 2.89× применяется при первом ACK |
| `tp->snd_cwnd` | Начальное окно перегрузки, вычисленное через `kcc_bdp()` при нейтральном усилении |

Защитный нижний предел в `kcc_update_bw` предотвращает перезапись введённой оценки первыми несколькими RTT выборок с низкой скоростью доставки во время STARTUP. Защита полной BW в `kcc_check_full_bw_reached` предотвращает преждевременное завершение STARTUP из-за обмена управляющими сообщениями iperf3.

### Коэффициент Десертной Скидки

Эффективная скорость инжекции:

```
coeff = (discount_ratio) / high_gain
      = (num / den) / 2.89
```

где `high_gain ≈ 2.89` — множитель пакетирования BBR STARTUP.

| num | coeff  | характеристика |
|-----|--------|----------------|
|  35 | 12.1%  | максимальная безопасность, худший путь |
|  50 | 17.3%  | центральная ось (по умолчанию) |
|  75 | 25.9%  | математическая оптимальная точка десерта |
|  80 | 27.6%  | математический потолок скорости (не должен превышаться) |

**Примечание:** `tcp_write_xmit` устанавливает начальный CWND равным `TCP_INIT_CWND` (10 сегментов, ≈15 КБ) для каждого нового соединения. CWND растёт только при поступлении удалённых ACK, поэтому десертная скорость является верхней границей скорости пакетирования — фактическая пропускная способность ограничена CWND до получения достаточного количества ACK для открытия окна.

### Конфигурация

Включение через `sysctl`:

```bash
sysctl -w net.kcc.kcc_kf_enable=0           # главное включение (по умолчанию 0)
sysctl -w net.kcc.kcc_kf_discount_num=50   # числитель десертной скорости (по умолч. 50, диапазон 35–75)
```

**Ключевые параметры sysctl** (`/proc/sys/net/kcc/`):

| Параметр | По умолч. | Диапазон | Описание |
|-----------|---------|-------|-------------|
| `kcc_kf_enable` | 0 | 0–1 | Главное включение глобальной инжекции Kalman BDP |
| `kcc_kf_discount_num` | 50 | 0–100 | Числитель десертной скорости (% от справедливой доли BW) |
| `kcc_kf_discount_den` | 100 | 1–100000 | Знаменатель десертной скорости |
| `kcc_kf_steady_mode` | 0 | 0/1 | — | Стационарный режим: при включении использует монотонный пик (kf_x_steady) для init_bw, игнорируя временные падения оценки KF |
| `kcc_kf_startup_r_pct` | 20 | 1–100 | R% шума измерений на фазе запуска |
| `kcc_kf_steady_r_pct` | 5 | 1–100 | R% шума измерений в установившемся режиме |

При включении `kcc_kf_steady_mode` (1) начальная пропускная способность для новых соединений использует монотонно возрастающий пик оценки KF (kf_x_steady) вместо текущей оценки, которая могла снизиться после завершения последнего высокоскоростного соединения. Это предотвращает голодание холодного старта на стабильных путях. Пик сбрасывается в ноль при отключении режима, обеспечивая чистый старт при повторном включении.

### Производительность в Первую Секунду (Транстихоокеанский, 212 мс RTT)

```
Без KF:  2.8 Mbps  →  85 Mbps  →  622 Mbps  →  установившийся
С KF:    50 Mbps   →  530 Mbps  →  650 Mbps  →  установившийся
```

Скорость в первую секунду прыгает с ~3 Мбит/с (холодный старт) до ~50 Мбит/с (десертный старт), а сходимость к установившемуся состоянию достигается за 2–3 секунды. Ретрансмиссии остаются нулевыми на протяжении всего процесса.

### Как Это Работает

1. Работающее соединение KCC входит в крейсерскую фазу PROBE\_BW → граница начала раунда → подаёт `kcc_kf_update(bw, 5%)` с текущей выборкой скорости доставки.
2. Фильтр Калмана обновляет свою оценку `kcc_kf_x` (скользящее среднее установившейся пропускной способности узкого места).
3. При открытии **нового** соединения `kcc_init` вызывает `kcc_kf_get_init_bw(sk)`, которая возвращает `fair × discount / high_gain` — скомпенсированную по усилению начальную оценку пропускной способности справедливой доли.
4. Эта оценка инициализирует `sk_pacing_rate`, `tp->snd_cwnd` и трекер пропускной способности `minmax` — соединение стартует на десертной скорости, а не с нуля.
